# Схема доставки dotnet-скиллов в жизненный цикл superpowers

**Дата:** 2026-07-04
**Статус:** дизайн одобрен с двумя поправками (полные описания в каталоге; нативная установка LSP-плагина) и требованием мультисервисного воркспейса

## Проблема

Два несвязанных набора скиллов:

- **superpowers** (`~/.claude/plugins/cache/claude-plugins-official/superpowers/6.1.1/`) — 14 процессных скиллов, ведущих разработку по этапам: brainstorm → план → исполнение (в т.ч. субагентами) → ревью → завершение ветки. Установлен как плагин, всегда в контексте.
- **dotnet-skills** (`/home/stsiapan/devTools/skills/dotnet-skills/`) — официальный репозиторий dotnet/skills: **98 технических скиллов в 16 плагинах** (test — 20, msbuild — 18, blazor — 9, maui — 8, diag — 7 и т.д.). Полная установка плагинами стоит ~18k токенов описаний в каждой сессии; полное содержимое ~1.1 МБ (~300k токенов). Скиллы ссылаются друг на друга по имени («use test-gap-analysis») и имеют 107 файлов в `references/`.

Требуется: каждый dotnet-скилл достижим с любого этапа superpowers (включая субагентов), постоянная цена в токенах близка к нулю, срабатывание автоматическое.

## Ограничения

1. **Контекст выживает.** Постоянная нагрузка ~0; полное знание грузится только в момент необходимости.
2. **Доступ соразмерен запросу.** Справка в основной сессии — дёшево, без спавна агента. Делегированный кусок работы — знание уезжает с ним, минуя основную сессию.
3. **Срабатывает само.** Без ручного «открой такой-то скилл».

## Рассмотренные подходы

**A. Роутер + генерируемый каталог + план как транспорт (выбран).** Один тонкий скилл-шлюз постоянно в контексте; каталог с полными описаниями всех скиллов — по требованию; полные скиллы — прямым `Read` по пути; к субагентам знание едет ссылками, вшитыми в план. Единственное исключение из «ничего не ставим» — нативная установка LSP-плагина `dotnet`: языковой сервер через `Read` не доставляется.

**B. Выборочная установка плагинов по проекту.** Включать в .NET-репозитории только нужные плагины (`/plugin install dotnet-test@...`). Отвергнут: гранулярность — плагин, а не скилл (dotnet-test один тащит 20 описаний); в .NET-проекте — а работа именно там — постоянная цена всё равно тысячи токенов; не решает доставку субагентам вообще.

**C. Библиотекарь-субагент или MCP-сервер знаний.** Отвергнут: справка из одного скилла стоит спавн агента или инфраструктуру сервера (нарушение ограничения 2); описания MCP-инструментов сами занимают контекст; пересказ агентом теряет точность — а `Read` файла и так атомарен и дёшев. RAG/эмбеддинги избыточны: 98 позиций помещаются в индекс на ~9k токенов, который модель сканирует напрямую и детерминированно.

## Архитектура (подход A)

Четыре компонента. Файлы superpowers не патчатся — интеграция живёт поверх, обновления обоих наборов безопасны.

### 1. Роутер — единственный постоянный житель контекста

`~/.claude/skills/dotnet-router/SKILL.md` — личный скилл (работает во всех проектах).

- **Description (~60–80 токенов, всегда в контексте)** — сильные триггеры: .NET, C#, csproj, sln, ASP.NET Core, EF Core, Blazor, MAUI, MSBuild, NuGet, xUnit/MSTest, dotnet CLI — «на любом этапе superpowers: brainstorm, план, исполнение, ревью, отладка».
- **Тело (~1–1.5k токенов, грузится только при вызове)** — правила интеграции с этапами superpowers (см. «Потоки доставки») и указание прочитать каталог.

### 2. Каталог — оглавление по требованию

`~/.claude/skills/dotnet-router/CATALOG.md`, генерируется скриптом из frontmatter всех 98 скиллов.

- Одна запись на скилл: `имя — полный description из frontmatter (включая USE FOR / DO NOT USE FOR) — абсолютный путь к SKILL.md`. **Описания не сжимаются**: негативные триггеры (DO NOT USE FOR) — это дизамбигуация между спутываемыми соседями (в dotnet-test 20 скиллов, в msbuild 18), оплаченная авторами набора; её потеря — главный риск неверной маршрутизации. Каталог грузится по требованию и только в .NET-сессиях — экономить на нём нечего.
- Итоговый размер ~8–9k токенов (24k символов описаний + имена и пути), санитарный потолок в генераторе — 15k.
- Группировка по доменам (test, msbuild, aspnetcore, …) + карта «этап superpowers → релевантные секции» (отладка → dotnet-diag; ревью → test-anti-patterns, assertion-quality, …).
- Строка с датой генерации и путём к генератору — видно устаревание.
- Каталог разрешает кросс-ссылки dotnet-скиллов друг на друга: имя → путь.

### 3. Генератор

`/home/stsiapan/devTools/skills/tools/gen-dotnet-catalog.sh` — парсит frontmatter, переносит описания дословно, проверяет существование путей, считает размер каталога (fail при >15k токенов). Запускается после `git pull` в dotnet-skills.

### 4. Хук — детерминированный триггер

SessionStart-хук в `~/.claude/settings.json`: если в дереве от директории запуска найден `*.sln` / `*.csproj` / `global.json` — инжектирует одну строку: «.NET repository detected — invoke dotnet-router skill before .NET work». В не-.NET-деревьях — ноль токенов.

Поиск рассчитан на воркспейсы, где запуск идёт на несколько уровней выше решений, а sln-файлы лежат в десятке веток на глубине 2–3+ уровней: глубина до 6, отсечение тяжёлых директорий, выход по первому совпадению:

```bash
find . -maxdepth 6 \( -name .git -o -name node_modules -o -name bin -o -name obj \) -prune \
  -o \( -name '*.sln' -o -name '*.csproj' -o -name global.json \) -print -quit
```

### 5. LSP-плагин — единственная нативная установка

Плагин `dotnet` (1 скилл + Roslyn language server) ставится по-настоящему: языковой сервер невозможно доставить через `Read`. Постоянная цена — одно описание скилла (~150–200 токенов); взамен этап исполнения получает компиляторную навигацию по C# вместо grep. Его `lsp.json` запускает Roslyn с `--autoLoadProjects` — проекты подгружаются по открываемым файлам, а не одним sln из корня, что совместимо с мультирешёночным воркспейсом (проверяется на смоуке: п. 4 раздела «Проверка»). Остальные 15 плагинов нативно не ставятся — живут в каталоге.

Установка **опциональна**: ядро схемы (роутер, каталог, план-транспорт, хук) от LSP не зависит. Недоступен .NET 10 SDK, нет CLI-команд установки, Roslyn прожорлив на связанных сервисах — плагин пропускается или выключается без ущерба для остального.

## Потоки доставки по этапам

| Этап | Механизм | Кто платит токенами |
|---|---|---|
| brainstorming, справка в основной сессии | роутер → каталог → `Read` нужного SKILL.md (один вызов, ~3k) | основная сессия, только за использованное |
| writing-plans | планировщик держит каталог; каждой задаче плана — блок `**Skills:**` с путями к 1–3 SKILL.md; в шапку плана — путь к каталогу | основная сессия, разово |
| subagent-driven-development, dispatching-parallel-agents | **push:** диспетчер механически копирует `**Skills:**` задачи в секцию Context промпта («Before starting, Read these skill files: …»). **pull:** в каждый промпт — строка «каталог .NET-скиллов: <путь>, обратись при непокрытой проблеме» | субагент, в своём контексте; основная сессия — ноль |
| executing-plans (отдельная сессия) | ссылки уже вшиты в файл плана — переживают границу сессии без участия основной | исполняющая сессия |
| requesting-code-review | стандартный .NET-ревью-набор (из тела роутера) добавляется путями в промпт ревьюера | ревьюер-субагент |
| systematic-debugging | карта этапов в каталоге: отладка → секция dotnet-diag (7 скиллов) | по месту отладки |
| finishing-a-development-branch | не требуется | — |

Ключевой ход: **план — транспорт знания.** Аннотирование происходит один раз, на этапе планирования, когда каталог уже в контексте; все последующие этапы (в этой сессии, в параллельной, у субагентов) получают маршрутизацию механическим копированием из артефакта, без повторных решений.

## Мультисервисный воркспейс

Рабочая раскладка: Claude Code запускается из директории на несколько уровней выше решений; вглубь идёт до десятка веток со своими `*.sln` на глубине 2–3+ уровней; сервисы связаны, работа часто затрагивает несколько одновременно.

Слои роутер/каталог/план от раскладки не зависят (абсолютные пути к файлам скиллов). Раскладка влияет на три места:

1. **Детекция** — хук ищет глубоко с отсечением и ранним выходом (см. компонент 4).
2. **Скоупинг задач** — dotnet-скиллы выполняют команды (`dotnet build`, `dotnet test`) относительно текущей директории, поэтому каждая задача обязана знать свой сервис. Правило в теле роутера: в мультирешёночном воркспейсе каждая задача плана получает блок `**Service root:**` — путь к директории sln, которой она принадлежит; задача, затрагивающая несколько сервисов, перечисляет все и помечает основной. Диспетчер копирует основной service root в слот `Work from:` промпта имплементера (слот уже существует в шаблоне superpowers — своих шаблонов не вводим). Команды dotnet выполняются из service root, не из корня воркспейса.
3. **LSP** — Roslyn с `--autoLoadProjects` грузит проекты по открываемым файлам, к корневому sln не привязан; поведение и память при работе сразу в нескольких сервисах проверяются на смоуке.

## Автоматика: три избыточных слоя

1. **Детерминированный:** SessionStart-хук — не зависит от внимания модели.
2. **Модельный:** description роутера в постоянном списке скиллов; дисциплина using-superpowers («1% вероятности → MUST invoke») уже навязана каждой сессии.
3. **Структурный:** после аннотирования плана срабатывание ниже по конвейеру не требуется — знание везут артефакты (план, промпты), а не память.

## Бюджет токенов

| Статья | Стоимость |
|---|---|
| Постоянно, любая сессия | ~250–300 (description роутера + скилл LSP-плагина) |
| Постоянно, .NET-воркспейс | +~20 (строка хука) |
| Первое .NET-действие в сессии | разово ~10k (тело роутера ~1–1.5k + каталог ~8–9k) |
| Каждый применённый скилл | ~3k в среднем, в контексте того, кто применяет |
| Для сравнения: полная установка плагинами | ~18k постоянно + размывание выбора |

## Обработка отказов

- **Каталог устарел** (dotnet-skills обновился): дата генерации в шапке каталога; регенерация после pull. Битые пути ловит проверка в генераторе.
- **Роутер не сработал:** страхует хук; хук не сработал (нестандартная структура репо) — страхует триггерная фраза description.
- **Субагенту не хватило пushed-скиллов:** pull-строка с путём каталога есть в каждом промпте.
- **Задача выполняется не в том сервисе:** блок `Service root:` обязателен для каждой задачи .NET-плана — правило в теле роутера; отсутствие блока — дефект плана, ловится ревью плана.

## Проверка (при имплементации)

1. Генератор: 98 записей в каталоге, описания перенесены без потерь, все пути существуют, размер ≤15k токенов.
2. Смоук в мультисервисном воркспейсе (запуск из корня, sln на 2–3 уровне): новая сессия → хук инжектировал строку, время поиска приемлемое; запрос «спланируй фичу на ASP.NET Core» → роутер вызван без ручного указания, каталог прочитан, задачи плана содержат блоки `**Skills:**` и `**Service root:**`.
3. Смоук делегирования: промпт имплементера содержит пути скиллов задачи, pull-строку и `Work from:` с service root задачи.
4. Смоук LSP: плагин `dotnet` установлен; из корня воркспейса открытие `.cs` в двух разных сервисах даёт корректную навигацию, потребление памяти терпимое.

## Объём имплементации

1. `gen-dotnet-catalog.sh` + сгенерированный `CATALOG.md`.
2. `~/.claude/skills/dotnet-router/SKILL.md` (description + тело: правила этапов и правило Service root).
3. SessionStart-хук в `~/.claude/settings.json` (глубокий поиск с отсечением).
4. Установка плагина `dotnet` из маркетплейса dotnet/skills (LSP).
5. Прогон проверок из раздела «Проверка».

Вне объёма: модификация superpowers или dotnet-skills, MCP-серверы, автосинхронизация dotnet-skills по расписанию (YAGNI — регенерация по требованию), нативная установка остальных 15 плагинов.
