# Дизайн: мультипрофильный `~/.claude` — ваниль + super в раздельных каталогах

**Дата:** 2026-07-07
**Статус:** утверждён (brainstorming), готов к написанию плана
**Автор:** Claude (Opus 4.8) + Stsiapan

## Проблема

Сейчас `setup.sh` владеет `~/.claude` целиком, поэтому **каждая** сессия `claude` тянет
полный superpowers-стек: плагин superpowers (форс-хук `You have superpowers` + 14 скиллов),
detect-dotnet хук, MCP-набор, codex-наж в `CLAUDE.md`, `effort: xhigh`. Для сложной работы
это то что нужно, но для быстрых мелочей — накладной расход и разбавление контекста. Хочется
иметь **выбор**: обычный `claude` = лёгкая ваниль, а полный пайплайн — по явному вызову.

## Цель

Держать на одной машине несколько **профилей** конфигурации, каждый в своём каталоге, и
переключаться между ними одной командой:

- `claude` → **vanilla** (`~/.claude`) — лёгкая приятная база без плагинов/хуков.
- `claude-super` → **super** (`~/.claude-super`) — сегодняшний полный стек, байт-в-байт.

Механизм расширяемый: будущий профиль (например `gsd`) добавляется одной записью в список
рецептов, а не переписыванием скрипта. `setup.sh` спрашивает, какие профили ставить, и
«внутри знает» рецепт каждого.

## Ключевой проверенный факт (обоснование архитектуры)

Проверено вживую на `claude 2.1.201`:

- **Плагины/хуки резолвятся из каталога конфига на старте сессии.** Отдельный
  `CLAUDE_CONFIG_DIR` → полностью изолированный профиль (свои плагины, хуки, MCP, скиллы,
  CLAUDE.md). Пустой каталог → чистая ваниль (нет инъекции `You have superpowers`).
- **`--settings enabledPlugins:false` НЕ выгружает** уже включённый плагин за сессию — значит
  «один каталог + флаги» чистую ваниль не даёт. **Отдельные каталоги — дают.**
- **`--bare` рвёт OAuth-авторизацию** (`Not logged in`; читает только `ANTHROPIC_API_KEY`),
  поэтому как ваниль для подписки не годится.
- **Авторизация шарится симлинком:** `.credentials.json` из второго каталога → симлинк на
  `~/.claude/.credentials.json`, перелогин не нужен (проверено — сессия залогинилась).
- `CLAUDE_CONFIG_DIR` в `--help` не задокументирован, но на 2.1.201 работает штатно.

## Ограничения

- **Идемпотентность.** Повторный `setup.sh` не даёт diff ни в одном каталоге и выходит 0
  (инвариант репозитория).
- **Профили независимы.** Падение раскатки одного профиля не рушит остальные (`|| warn`).
- **Публичный репозиторий.** Никаких секретов в git. Выбор профилей и их флаги — в
  `~/.config/claudefiles/secrets.json` (вне git, `600`), как сейчас.
- **Не трогать файлы superpowers/dotnet-skills.** Профили только компонуют то, что уже есть.
- **Тестируемость.** Всё новое — против фейкового `claude` и фикстур-`$HOME`, как в репо.
- **`model`/`effortLevel` в ваниль не управляются** — пользователь ставит их сам.

## Не-цели (осознанный YAGNI)

- Профиль `gsd` сейчас **не делаем** — только закладываем расширяемость.
- Декларативный манифест `profiles.json` (подход B) — оверинжиниринг для 2 профилей;
  оставлен как путь миграции, если профилей станет много.
- Allowlist разрешений в ваниль — не нужен: пользователь всегда в auto mode.
- Единая история между профилями — по умолчанию раздельная (изоляция); симлинк опционален.
- Запуск главной сессии «как агента» (`--agent`) — это подмена персоны, ортогонально
  загрузке обвеса; профилям не нужно.

## Профили и каталоги

| Профиль | Каталог | Вызов | Содержимое |
|---------|---------|-------|------------|
| **vanilla** | `~/.claude` (дефолт) | `claude` | статуслайн, личный CLAUDE.md, Context7 MCP, светлая тема; без плагинов/хуков |
| **super** | `~/.claude-super` | `claude-super` | сегодняшний полный стек, байт-в-байт |

`vanilla` ставится всегда. `super` — по выбору. Новый профиль → `~/.claude-<name>` + запись
в списке рецептов.

## Архитектура

### Механизм целевого каталога

Вводится одна переменная **`CLAUDEFILES_TARGET`** (по умолчанию `$HOME/.claude`). Все
`*_apply`-функции пишут в неё вместо жёсткого `$HOME/.claude`:

- `settings.sh`, `skills.sh` — правятся (сейчас хардкодят `$HOME/.claude`).
- `claudemd.sh` — уже готов (override-паттерн `CLAUDEFILES_CLAUDE_MD` есть).
- Фазы `plugins`/`mcp` зовут `claude` — на время фазы экспортируется
  `CLAUDE_CONFIG_DIR="$CLAUDEFILES_TARGET"`, чтобы CLI ставил в нужный каталог.

Значение по умолчанию сохраняет текущее поведение для кода, не знающего о профилях.

**MCP — пер-профиль.** Каждый рецепт передаёт свой набор серверов (ваниль → `context7`;
super → его флаги), а манифест сверки `managed-mcp.json` становится **пер-каталог**
(`managed-mcp.<profile>.json` или файл внутри target-каталога), чтобы `mcp_apply` убирал/добавлял
ровно серверы этого профиля и не трогал чужие. Иначе один манифест на машину «перетирал» бы
наборы разных профилей.

### Слой рецептов (подход A)

Профиль = маленькая bash-функция-рецепт, зовущая существующие `*_apply` с целевым каталогом
и своим набором флагов. Список профилей — массив. Псевдокод оркестратора:

```
config:   спросить, какие профили (vanilla всегда; super y/N; при super — его флаги как сейчас)
for профиль in выбранные:
    export CLAUDEFILES_TARGET=<каталог профиля>
    recipe_<профиль>            # зовёт нужные *_apply
ensure_credentials_symlink      # для не-дефолтных каталогов
generate_aliases                # ~/.config/claudefiles/profile-aliases.sh
readiness_report                # по каждому профилю
```

Новый профиль = функция `recipe_<name>` + строка в массиве. Файл-модуль: `lib/profiles.sh`.

### Компонент — рецепт vanilla (`~/.claude`)

- `settings.json`: `theme: light`, `statusLine` → абсолютный путь к скрипту статуслайна.
  **`model`/`effortLevel` не входят в управляемые ключи ванили** — не перетираются.
- **Статуслайн** — новый `claude/statusline/statusline.sh` (в git). Выводит: базовое имя
  каталога, git-ветку + `*` если рабочее дерево грязное, имя модели, индикатор остатка
  контекста. Палитра подобрана под светлую тему. Читает JSON статуса со stdin (контракт
  Claude Code statusLine). Модуль переиспользуемый — super сможет включить позже.
- **Личный `CLAUDE.md`** — marker-блок (по образцу `claudemd.sh`) с личным стилем: коротко и
  прямо, без эмодзи, без em-dash, избегать «ensure/leverage/robust/seamless/utilize»,
  цитировать `file:line`, комментарии в коде — только по явной просьбе. Отдельный блок,
  редактируется свободно; управляется идемпотентно.
- **Context7 MCP** — рецепт ванили драйвит **фиксированный** MCP-набор (только `context7`),
  независимо от флагов super; опциональный `context7_api_key` спрашивается один раз (как
  сейчас, общий на профили).
- Плагинов, superpowers, форс-хуков, detect-dotnet — **нет**.

### Компонент — рецепт super (`~/.claude-super`)

Ровно сегодняшний вывод `setup.sh`, но с `CLAUDEFILES_TARGET=$HOME/.claude-super`:
superpowers (всегда), dotnet/codex по флагам, detect-dotnet хук, MCP-набор, codex-наж в
CLAUDE.md, `model: opus[1m]`, `effort: xhigh`, `theme: dark`, `tui: fullscreen`. Флаги super —
это существующие флаги (`dotnet_skills`, `codex_review`, `context7`, `playwright`, `azure_mcp`,
`ado`), теперь понимаемые как «флаги профиля super».

### Компонент — авторизация и история

- `ensure_credentials_symlink`: для каждого не-дефолтного каталога создаёт
  `<dir>/.credentials.json` → симлинк на `$HOME/.claude/.credentials.json`, если ещё нет.
  Дефолтный `~/.claude` держит реальный файл (туда пишет `claude login`). Идемпотентно.
- **История — раздельная** (свой `history.jsonl` на каталог). Осознанно: контекст super не
  мешается с ванилью. Единую историю при желании даёт такой же симлинк — не в MVP.

### Компонент — переключение (алиасы)

`setup.sh` генерит `~/.config/claudefiles/profile-aliases.sh` (вне git, идемпотентно):

```bash
alias claude-super='CLAUDE_CONFIG_DIR="$HOME/.claude-super" claude'
```

Один алиас на каждый не-дефолтный установленный профиль. `setup.sh` печатает строку для
подключения: `source ~/.config/claudefiles/profile-aliases.sh` (или её подхватывает
dotfiles-репо через chezmoi). `claude` без env = vanilla.

## Миграция существующего `~/.claude`

Сейчас `~/.claude` = полный super. После первого прогона нового `setup.sh`:

- Vanilla-рецепт применяется к `~/.claude`: `settings.json` через `jsonmerge.py` заменяет
  управляемые ключи на ванильные (`enabledPlugins` без superpowers, `hooks` пустой) → инъекция
  `You have superpowers` и detect-dotnet хук **исчезают сами**.
- Super ставится заново в `~/.claude-super`.
- Файлы плагинов в `~/.claude/plugins` остаются на диске (безвредно). Опциональная подчистка —
  не в MVP.

Явный тест миграции: фикстура-`$HOME` с «сегодняшним» `~/.claude` → после setup в `~/.claude`
нет `enabledPlugins.superpowers` и нет хука, а `~/.claude-super` — полный.

## Обработка ошибок и идемпотентность

- Раскатка каждого профиля независима; ошибка одного → `warn`, остальные продолжают (как
  сейчас `plugins_apply` под `|| warn`).
- Повторный `setup.sh` = ноль diff во всех каталогах, выход 0.
- Симлинк creds и генерация алиасов — идемпотентны (создать-если-нет / переписать-детерминированно).
- Нет TTY при первом конфиге → падение с понятным списком (как сейчас), не зависание.

## Тестирование

По паттерну репо (`skills/tools/test-*.sh` против фейкового `claude` и фикстур-`$HOME`):

- `settings.sh`/`skills.sh` уважают `CLAUDEFILES_TARGET` (пишут в заданный каталог).
- Фазы plugins/mcp экспортируют `CLAUDE_CONFIG_DIR` = target (фейк-`claude` фиксирует, куда звали).
- Меню профилей: vanilla всегда, super по флагу.
- Vanilla-рецепт: в `settings.json` есть `theme:light` + `statusLine`, нет `enabledPlugins`/
  `hooks`; `CLAUDE.md` содержит личный блок; `model`/`effort` не перетёрты.
- Super-рецепт: полный стек оказывается в `~/.claude-super`, а не в `~/.claude`.
- Идемпотентность симлинка creds; генерация алиасов (ровно один алиас на не-дефолтный профиль).
- Миграция «был super → стал vanilla» в `~/.claude`.
- Инвариант: повторный прогон без diff (расширить существующий `test-setup-idempotent.sh`).

## Что в git / вне git

- **В git:** `lib/profiles.sh`, правки `lib/settings.sh`/`lib/skills.sh`/`setup.sh`, шаблоны
  settings (vanilla + super), `claude/statusline/statusline.sh`, текст личного CLAUDE.md-блока,
  тесты, README.
- **Вне git (генерится/машинно-специфично):** сами каталоги профилей, `secrets.json`,
  dotnet-каталог, creds-симлинки, `profile-aliases.sh`.

## Раскладка (дельта к текущей)

```
lib/profiles.sh               # НОВОЕ: список профилей + recipe_vanilla / recipe_super
lib/settings.sh               # правка: писать в $CLAUDEFILES_TARGET
lib/skills.sh                 # правка: писать в $CLAUDEFILES_TARGET
setup.sh                      # правка: меню профилей, цикл по рецептам, creds-симлинк, алиасы
claude/settings/              # settings.vanilla.template.json + settings.super.template.json (был settings.template.json)
claude/statusline/statusline.sh   # НОВОЕ
skills/tools/test-*.sh        # новые/расширенные тесты
```
