# Дизайн: документация проекта и диаграммы на Mermaid

**Дата:** 2026-07-14
**Статус:** утверждён (brainstorming), готов к написанию плана
**Автор:** Claude (Opus 4.8) + Stsiapan
**Связано:** STS-73 (реализация), STS-71 (спайк выбора формата)

## Проблема

Проект пишется через ИИ, архитектура не держится в голове. Единственная существующая
тех-диаграмма (`docs/architecture.md`, создана в спайке STS-71) мешает в одном flowchart
четыре среза сразу: фазы `setup.sh`, фан-аут на профили, модули `lib/*_apply` и хранилища
данных. Плюс зависимости через env (`CLAUDEFILES_TARGET`/`CLAUDE_CONFIG_DIR` экспортятся в
субшелле, profiles.sh:73) приходится тянуть длинными стрелками через весь граф. Итог:
пересечения, нечитаемо, не production-ready.

Смежная проблема: поведение живой сессии (`claude`/`claude-super`) и гарантии, которые держит
тест-сьют, не задокументированы нигде.

## Цель

Набор фокусных диаграмм и доков, по которым систему можно понять не читая код, и который
ИИ удобно поддерживать. Каждая диаграмма отвечает на один вопрос. Диаграммы живут рядом с
кодом в `docs/`, правятся в том же PR, что и код.

## Принципы читаемости

Это ядро дизайна, не косметика.

1. **Одна диаграмма = один вопрос.** Не смешивать статику, динамику, данные и runtime.
2. **Правильный тип под срез.** Динамику (прогон) рисуем `sequenceDiagram` — у него нет
   пересечений по построению. Статику (зависимости) — слоёными `subgraph` со стрелками только
   вниз.
3. **Схлопывать фан-аут.** Ребро в `subgraph`-кластер вместо N рёбер в его узлы. Вездесущие
   зависимости (`common.sh`) не рисуем 8 стрелками, а помечаем подписью «доступен всем слоям».
4. **Бюджет узлов.** ≤ ~12 узлов на диаграмму. Больше — делить.
5. **Одно направление раскладки** на диаграмму (TD или LR, не смешивать).
6. **Обратные рёбра-исключения** помечать пунктиром и подписью (единственное такое:
   `_chromium_present`→`config`, deps.sh:58).

## Конвенции Mermaid

- **Без `<br/>`** в лейблах: рендерер Mermaid в Linear на нём падает (проверено на STS-71, см.
  память `linear-mermaid-br`). Многострочность — скобки или запятая. Правило безопасно и для
  GitHub/Obsidian.
- **Один срез = одна фокусная вьюха** под своим H2-заголовком. Файл может держать несколько
  диаграмм (`architecture.md` держит 4), но каждая отвечает на один вопрос, не сборная.
  Структура `architecture.md` задана жёстко ниже — без свободного «ещё блок сюда».
- **Подсветка обсуждаемого** — класс `new` (`classDef new ...` + `:::new`), снимается при мёрже.
- Рендер каждой диаграммы проверяется через `mmdc` перед коммитом.

## Набор диаграмм (6)

Все обоснованы разбором кода (граф зависимостей, слои L0-L4, runtime vs deploy).

### 1. Компонентная по слоям (статика)
- **Вопрос:** какие модули есть и как зависят.
- **Тип:** `flowchart TD`, слои как `subgraph`, стрелки вниз.
- **Слои по ответственности:** оркестрация (`setup.sh`, `profiles.sh`) → системные deps
  (`deps.sh`) → модули-appliers (`settings.sh` + хелпер `jsonmerge.py`, `skills.sh`,
  `plugins.sh`, `mcp.sh`, `claudemd.sh`) → хранилище конфига (`config.sh` + `config_io.py`,
  владеет `secrets.json`) → утилиты (`common.sh`).
- **Важно:** `jsonmerge.py` (jsonmerge.py:2) — логика мёрджа settings, вешается на `settings.sh`,
  а НЕ в конфиг-слой; `config_io.py` (config_io.py:2) — хранилище секретов. Не смешивать.
- **Обоснование:** слои и рёбра из графа зависимостей; back-edge deps.sh:58 помечен.
- **Файл:** `docs/architecture.md`.

### 2. Data-flow хранилищ
- **Вопрос:** что откуда читается и куда пишется при деплое.
- **Тип:** `flowchart LR`.
- **Содержимое:** источники (шаблоны `claude/settings/*.template.json`, `secrets.json`) →
  модули → домашние стора (`settings.json`, `managed-mcp.<profile>.json`, `CLAUDE.md`,
  wrapper `~/.local/bin/claude-<name>`, симлинк `.credentials.json`).
- **Обоснование:** таблица data-stores из разбора (пути с file:line).
- **Файл:** `docs/architecture.md`.

### 3. Деплой как sequenceDiagram (динамика)
- **Вопрос:** что происходит за один прогон `setup.sh`.
- **Тип:** `sequenceDiagram`.
- **Содержимое:** preflight → config (`config_ensure_all`) → deps → `provision_selected` →
  `loop` по профилям в субшелле (export TARGET/CONFIG_DIR) → `recipe_*` → модули → verify.
- **Пути деградации (обязательно, не только happy path):** нет `claude` на preflight — warn,
  не фатал (setup.sh:8); deps не фатальны, warn+continue (deps.sh); провал профиля копится в
  `PROVISION_FAILED`, прочие профили доезжают, затем exit 1 (setup.sh:79). Рисуем через
  `alt`/`opt`-фрагменты, чтобы не раздувать.
- **Обоснование:** setup.sh:7-82, provision_selected profiles.sh:66-80.
- **Файл:** `docs/architecture.md`.

### 4. Recipe-diff vanilla vs super
- **Вопрос:** чем отличается провижн двух профилей.
- **Тип:** `flowchart LR`, две колонки-`subgraph`.
- **Содержимое:** упорядоченные `*_apply` с флагами; vanilla опускает `plugins_apply` и
  self-verify; передаёт `_mcp_legacy` 3-м аргументом; personal/codex блоки `CLAUDE.md`
  инвертированы.
- **Обоснование:** recipe_vanilla profiles.sh:6-14, recipe_super profiles.sh:16-36.
- **Файл:** `docs/architecture.md`.

### 5. Runtime-сессия
- **Вопрос:** что работает в живой сессии `claude`/`claude-super`.
- **Тип:** `flowchart TD`.
- **Содержимое, с явным разделением общего и профиль-специфичного:**
  - общее для обоих профилей: statusline (профиль-бейдж по arg/`CLAUDE_CONFIG_DIR`), звук на
    Stop/Notification (`paplay ready.wav` — есть в обоих шаблонах, vanilla:9 и super:20).
  - только super: SessionStart-хук `detect-dotnet.sh`, плагины, расширенный MCP-набор.
  - MCP: vanilla = только context7; super = флаг-гейтед набор.
- **Зачем разделение:** без него ваниль читается минимальнее, чем есть (звук у неё тоже есть).
- **Обоснование:** шаблоны settings, statusline.sh, detect-dotnet.sh, build_servers.py.
- **Файл:** `docs/runtime.md`.

### 6. User-flow (обновить существующую)
- **Вопрос:** что запускает пользователь и с каким исходом.
- **Тип:** `flowchart TD` (уже есть).
- **Содержимое:** chezmoi bootstrap vs прямой прогон → опрос фич → deps → провижн → исходы.
- **Файл:** `docs/user-flow.md` (оставить, привести к конвенциям).

## Набор доков (5)

- **`docs/architecture.md`** — обзор + диаграммы 1-4 + короткий текст по слоям и рецептам.
- **`docs/user-flow.md`** — диаграмма 6, без изменений структуры.
- **`docs/runtime.md`** — диаграмма 5 + поведение живой сессии, разница vanilla/super в runtime.
- **`docs/invariants.md`** — гарантии из тест-сьюта. Каждая ссылается на конкретный тест в
  `skills/tools/`. Маппинг гарантия → источник (проверить file:line при написании):
  - идемпотентность / zero-diff на повторный прогон → `test-setup-idempotent.sh`,
    `test-multiprofile.sh`, покомпонентно `test-mcp.sh`, `test-plugins.sh`, `test-claudemd.sh`.
  - `chmod 600` на `secrets.json` и MCP-манифестах → `test-config.sh`, `test-mcp.sh`.
  - секреты не в git → `test-secrets-not-tracked.sh`.
  - изоляция профилей (субшелл, без утечки env, отдельный target) → `test-profiles.sh`,
    `test-settings.sh`, `test-skills.sh`.
  - полнота и одноразовость миграции super→vanilla → `test-multiprofile.sh`, `test-settings.sh`.
  - неклоббер чужих артефактов (wrapper, creds) → `test-profiles.sh`.
  - whole-token матчинг (без substring-коллизий) → `test-deps.sh`, `test-plugins.sh`.
  - робастность statusline (инъекции/типы/санитайз) → `test-statusline.sh`.
- **`docs/config-schema.md`** — ключи `secrets.json` (флаги, `ado.orgs`/`ado.pat`,
  `context7_api_key`, `playwright.chromium_path`) плюс владение и семантика записи, не только
  чтение: prompt-once для флагов (config.sh:48), запись пустого для optional и почему это не
  переспрашивается (config.sh:44), `chmod 600` на каждой записи (config_io.py:40), present-empty
  vs absent, required vs optional, булевы как JSON-bool, легаси-строка `"true"` терпится в
  MCP-конфиге (build_servers.py:8-10). Обоснование: config.sh, config_io.py, build_servers.py.

## Структура файлов

```
docs/
  architecture.md    # диаграммы 1-4 + проза
  user-flow.md       # диаграмма 6
  runtime.md         # диаграмма 5 + проза
  invariants.md      # гарантии тест-сьюта
  config-schema.md   # схема secrets.json
```

Существующий `docs/architecture.md` и `docs/user-flow.md` (из спайка) переписываются под
финальный вид. `README.md` получает ссылки на новые доки в разделе «Раскладка».

## Рендер и проверка

Точная команда (chromium из кеша puppeteer, чтобы не тянуть свой):

```
PUPPETEER_EXECUTABLE_PATH=<cached-chrome> \
  npx -y @mermaid-js/mermaid-cli -i docs/<file>.md -o <out>.png -b white
```

Каждый `.md` с диаграммами прогоняется этой командой; exit-0 и число найденных диаграмм ==
ожидаемому — объективный гейт. Аккуратность раскладки — отдельная человеческая проверка по
рендеру. Проверка в Linear — вручную при переносе (ограничение `<br/>` уже в конвенции).
`make diagrams` — опционально позже, не блокер.

## Вне охвата (YAGNI)

- Отдельный переносимый docs-фреймворк (упомянут в спайке как «фундамент») — не сейчас.
- `make diagrams` и CI-рендер — опционально, не блокер.
- D2/PlantUML — решение спайка: Mermaid дефолт, D2 только если схема перерастёт авто-раскладку.

## Стадийность

Охват полный (утверждён), но план исполняем фазами, чтобы был рабочий промежуточный результат:
- Фаза 1: `architecture.md` (диаграммы 1-4) + `invariants.md`. Это чинит главную боль сразу.
- Фаза 2: `runtime.md` (диаграмма 5) + `config-schema.md` + освежить `user-flow.md` + ссылки в
  `README.md`.
Каждая фаза самодостаточна и коммитится отдельно.

## Критерий готовности

Объективные гейты (проверяемы командой):
- Все `.md` с диаграммами рендерятся `mmdc` с exit-0; число диаграмм == ожидаемому (6 всего).
- Ни одного `<br/>` в лейблах (grep пуст).
- `README.md` ссылается на все новые доки.

Человеческая проверка:
- 6 диаграмм читаемы: бюджет узлов ≤ ~12 соблюдён, пересечений нет или помечены как исключения.
- 5 доков написаны, диаграммы встроены, факты сверены с кодом (file:line).
